FourD

Web Component: Force Directed Graph Visualization

Usage

(Big fat disclaimer: This is not done yet, but could use some help to let this thing be its best)

FourD is intended as a plain javascript (to keep the build process simple) web component. To use it, include everything in the js folder, specifically the entry point js/fourd.js. Handling this manually is not recommended, use a bundler! parcel for example, because that’s how the folk at html5-boilerplate do it.

Long story short:

in index.html:

...

<style>

  jmm-fourd {

     width: 50vw;

     height: 75vh;

     background-color: lightgreen;

     --vertex-color: darkblue; (needs fixing)

     --edge-color: darkgray; (needs fixing, I think)

  }
</style>

<jmm-fourd id=“fourd”></jmm-fourd>

<script>

  import “./fourd.js”;

  var fourd = document.querySelector(‘#fourd’);

  window.fourd = fourd;

 

  var v1 = fourd.add_vertex(), v2 = fourd.add_vertex(), v3 = fourd.add_vertex(), v4 = fourd.add_vertex(); fourd.add_edge(v1, v2); fourd.add_edge(v2, v3); fourd.add_edge(v1, v3); fourd.add_edge(v1, v4); fourd.add_edge(v2, v4); fourd.add_edge(v3, v4); fourd.follow(1);

</script>

...

Another warning is in order. Don’t edit the files in docs. I know it’s faster than editing something and then compiling it, but the compile step consists of one of the npm run builds (either npm run github or npm run local for most people), setting up the toolchain is a onetime thing, and you don’t risk having your changes overwritten by the next build. In fact, the entire docs/ directory (and the .cache directory, just to be sure) gets overwritten with every build. Remove the line segment reading rm -rf docs/ .cache && if you want to keep the cache, for example to keep compile times lower.

If you just want to see the current page, head on over to https://joshuamoore.dev/fourd/ and open your devtools. It’s never too eary to learn what the stuff we look at is made of, and this knowledge will come in handy with the next myspace. But I digress. With the dev tools open, paste this short program into the Javascript Console:

var v1 = fourd.add_vertex(), v2 = fourd.add_vertex(), v3 = fourd.add_vertex(), v4 = fourd.add_vertex(); fourd.add_edge(v1, v2); fourd.add_edge(v2, v3); fourd.add_edge(v1, v3); fourd.add_edge(v1, v4); fourd.add_edge(v2, v4); fourd.add_edge(v3, v4); fourd.follow(1);

You’ll see that it doesn’t work. But that and various other improvements are what I’m working on right now. This is merely a snapshot for history. It’s half done, half unfinished, but organized in a way that allows me to come back to it over time and work on one aspect of the program at a time. Spheres in addition to cubes. Arrows in addition to edges. CSS properties to drive FourD behavior. The bones are there, the flesh takes a while to grow.

var fourd = document.querySelector(‘jmm-fourd’);

var id1 = fourd.add_vertex({shape: “cube” | “sphere”, color: “color name” | colorHex, size: Number (default 10), texture: “path/to/img” });

Specify either a texture or a color, the default is the color black.

var id2 = fourd.add_vertex();

var e = fourd.add_edge(id1, id2, {color: CSS Color, width: Number (default 1), strength: Number, (default 1.0)}?)

Optional option parameter, but it might not be optional yet.

The question marks mean I don’t know what the values for these should be yet, or whether it’s the fault of my implementation of the calculation that the vertices currently fly apart in a probably seizure inducing split, quite jarring stuff.
